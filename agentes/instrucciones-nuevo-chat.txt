Contexto esencial:
La app es una web estatica para controlar una vending machine. Actualmente guarda datos en `localStorage`.

La zona delicada es la contabilidad:
- La venta debe pertenecer al producto anterior (`previousProduct`).
- El stock futuro debe pertenecer al producto actual (`product`).
- La posicion fisica solo debe servir para comparar stock anterior contra encontrado.
- La vista puede fusionar productos repetidos, pero eso no debe afectar la contabilidad.

Decision recomendada:
No seguir parchando la contabilidad basada en slots.
Crear una capa explicita de movimientos contables:

`salesMovements`

Y una capa separada de stock futuro:

`nextStock`

Idea de registro futuro:

record = {
  date,
  fuelCost,
  slotsSnapshot,
  salesMovements,
  nextStock
}

Regla de oro:
Al guardar una visita, la app debe congelar las ventas como movimientos por producto. Los informes deben leer esos movimientos, no recalcular desde la vista.

No rompas estas reglas:
1. Meses cerrados no se editan.
2. Anos cerrados no se recalculan.
3. Cambios de precio solo afectan el mes actual en adelante.
4. El informe debe conservar toda la lista de productos, incluso con ventas 0.
5. El orden de productos es importante porque el usuario copia la columna a Excel.
6. La pantalla movil es prioridad.
