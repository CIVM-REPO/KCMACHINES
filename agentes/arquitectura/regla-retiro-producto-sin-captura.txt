REGLA: RETIRAR O CAMBIAR PRODUCTO CON STOCK ANTERIOR

Problema:
Si un espacio tenia stock anterior y el usuario cambia el producto a otro producto o a EMPTY antes de indicar cuanto encontro, la app no debe interpretar silenciosamente que todo se vendio.

Ejemplo:
Dia anterior:
- A01 tenia COCA-COLA.
- Stock anterior: 10.

Dia actual:
- El usuario quiere retirar COCA-COLA o cambiar A01 a AGUA.
- Todavia no ingreso "Encontre".

Riesgo:
Si "Encontre" queda en 0 por defecto, la contabilidad podria registrar 10 ventas de COCA-COLA aunque en realidad el usuario no capturo el dato.

Regla:
Cuando un slot tiene `previousStock > 0`, antes de cambiar `product` o guardar la visita debe existir captura explicita de "Encontre".

Resultado correcto:
1. La app avisa que primero debe indicar cuanto encontro.
2. La app abre el dialogo de captura del espacio.
3. Luego de capturar, el usuario puede cambiar el producto o dejarlo EMPTY.
4. La venta se calcula para `previousProduct`.
5. El stock futuro se calcula para `product` actual.

Ejemplo con retiro:
- Previous product: COCA-COLA.
- Previous stock: 10.
- Encontre: 4.
- Producto actual: EMPTY.

Resultado contable:
- Venta: 6 COCA-COLA.
- Stock futuro: no queda stock en A01.

Implementacion actual:
- `slotNeedsFoundCapture(slot)` detecta espacios con stock anterior y sin `foundCaptured`.
- `foundCaptured` marca que el usuario ya confirmo "Encontre"; `hasEntry` no basta porque tambien se usa para sincronizar productos repetidos.
- `updateSlotProduct()` bloquea cambios de producto si falta "Encontre".
- `saveSelectedDate()` valida capturas pendientes antes de construir el record.

Relacion con `salesMovements`:
Esta regla debe mantenerse cuando se migre a movimientos contables. Un movimiento de venta solo debe construirse desde una captura explicita, no desde un valor 0 inicial que el usuario nunca confirmo.
