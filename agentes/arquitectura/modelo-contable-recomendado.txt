MODELO CONTABLE RECOMENDADO: MOVIMIENTOS CONTABLES

Nombre del modelo:
Modelo por movimiento contable.

Idea:
Al guardar una visita, se calculan y congelan las ventas como movimientos por producto.
Desde ese momento, los informes usan esos movimientos y no vuelven a recalcular ventas desde la maquina visual.

Estructura recomendada para cada visita:

record = {
  date: "2026-05-28",
  fuelCost: 0,

  slotsSnapshot: [
    {
      slotCode: "A01",
      previousProduct: "COCA-COLA",
      previousStock: 10,
      found: 3,
      product: "AGUA AQUA",
      left: 8
    }
  ],

  salesMovements: [
    {
      product: "COCA-COLA",
      sold: 7,
      price: 1.00,
      cost: 0.55,
      sourceSlot: "A01"
    }
  ],

  nextStock: [
    {
      slotCode: "A01",
      product: "AGUA AQUA",
      stock: 8
    }
  ]
}

Responsabilidades:

`slotsSnapshot`
- Sirve para reconstruir la pantalla de esa visita.
- Guarda lo que el usuario vio/capturo.
- Puede incluir datos utiles de auditoria.

`salesMovements`
- Es la verdad contable.
- Cada movimiento pertenece a un producto.
- Guarda precio y costo del momento.
- Es lo que usan informes, cierres y Excel.

`nextStock`
- Es la verdad operativa para la proxima visita.
- Se guarda por espacio fisico.
- Usa el producto actual y la cantidad dejada.

Como maneja casos:

Primer refill:
- `previousStock` es 0.
- No se crean ventas.
- Solo se crea `nextStock`.

Producto repetido:
- Si un producto ocupa varios espacios, puede generar uno o varios movimientos.
- El informe suma movimientos por `product`.
- La UI puede capturar una sola vez y repartir/sincronizar, pero contabilidad no depende de la UI.

Cambio de producto:
- Venta se crea con `previousProduct`.
- Stock futuro se crea con `product`.

Cambio de posicion:
- No altera ventas ya guardadas.
- La proxima visita parte del `nextStock` guardado por espacio.

Empty:
- Si antes habia producto, puede generar venta.
- Si ahora queda empty, el `nextStock` de ese espacio queda vacio o no se incluye.

Cierre mensual:
- Suma `salesMovements` del mes.
- Congela resumen.
- Bloquea edicion del mes.

Cierre anual:
- Compacta meses cerrados.
- Guarda resumen anual.
- Conserva ultimo `nextStock` como base del siguiente ano.

Cambio de precio:
- Cada movimiento guarda precio/costo del momento.
- Meses cerrados no cambian.
- Mes actual desde el cambio en adelante usa el nuevo precio.

Ventajas:
- Reduce doble conteo.
- Evita recalcular contabilidad desde elementos visuales.
- Hace mas claro que es stock operativo y que es venta.
- Facilita pruebas.
- Facilita exportar a Excel.

Desventaja:
- Requiere migracion media.
- Hay que adaptar informes para leer `salesMovements`.
- Hay que mantener compatibilidad temporal con registros antiguos basados en `slots`.

Recomendacion:
Implementar este modelo antes de seguir agregando funciones contables nuevas.

Nota de implementacion:
La app ya puede guardar registros nuevos con `salesMovements` y `nextStock`, manteniendo `slots`/`slotsSnapshot` para pantalla y compatibilidad. Los informes deben seguir entrando por helpers compatibles para no romper registros antiguos que solo tengan `slots`.

Compatibilidad actual con productos repetidos:
Mientras la UI siga capturando productos repetidos una sola vez, `nextStock` debe respetar la captura primaria (`stockCounted`) para evitar triple conteo accidental. Si en el futuro se cambia a captura real por espacio fisico, entonces `nextStock` podra guardar una entrada por cada espacio con cantidad independiente.
