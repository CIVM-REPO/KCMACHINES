OBJETIVO DEL CAMBIO

Se refactorizo la logica principal para separar tres responsabilidades que antes estaban mezcladas:

1. Matriz visual de espacios fisicos.
2. Captura temporal de la visita actual.
3. Contabilidad historica por producto y maquina.

La matriz visual ya no debe ser la fuente directa de la contabilidad. La contabilidad ahora se basa en detalles por producto, usando la formula:

UV = stockAnterior - SM

Donde:
- SM = stock encontrado al llegar a la maquina.
- NS = nuevo stock dejado al terminar la visita.
- UV = unidades vendidas.


CAMBIOS PRINCIPALES EN LA APP

1. Matriz visual inicial vacia

Se elimino el layout estandar de productos preasignados.
Ahora la maquina inicia con todos los espacios en EMPTY cuando no hay datos previos.

Archivo principal:
- script.js

Cambio clave:
- baseProductLayout queda vacio.


2. Separacion visual/contable

Se agrego una capa temporal de visita basada en productos:

- currentVisitDetails
- visitDetails
- stockAnterior
- representative
- SM
- NS
- UV
- status
- note

Esto permite que la contabilidad no dependa de la posicion visual del producto.


3. Representante visual

Cada producto tiene como maximo un representante visual por visita.
El representante se calcula por barrido global:

1. De arriba hacia abajo.
2. De izquierda a derecha dentro de cada fila.

Solo el representante abre el dialogo de captura.
Los duplicados conservan selector de producto, pero no capturan datos contables directamente.


4. Productos repetidos y fusion visual

Se agrego fusion visual por fila.
Cuando un producto se repite dentro de una misma fila, se agrupa desde su primer ejemplar.

Esta fusion afecta la vista, no la contabilidad historica.


5. Fechas guardadas bloqueadas

Por seguridad, una fecha ya guardada queda bloqueada para edicion.
Tambien siguen bloqueados los meses cerrados.

Esto reduce riesgo de inconsistencias mientras se estabiliza el nuevo modelo.


6. Validaciones de captura

Antes de guardar una visita:

- Si un producto tiene representante, debe capturar SM y NS.
- Si un producto no tiene representante pero tiene stockAnterior > 0, la app solicita SM manualmente.
- Si SM > stockAnterior, se bloquea el guardado porque UV seria negativo.
- Si stockAnterior = 0 y no hay representante, se permite SM = 0, NS = 0, UV = 0.


7. Carga inicial

Si no hay visitas previas ni stock anterior, el primer guardado se marca como:

type = carga_inicial

En carga inicial no se generan ventas.
Solo se guarda el stock dejado como base para la siguiente visita real.


8. Cache de imagenes de productos

Se optimizo la carga de imagenes:

- Se dejo una version fija para las imagenes durante la sesion/build.
- Se precargan imagenes del catalogo una vez.
- Los rerenders de la maquina ya no fuerzan nuevas URLs con Date.now().

Archivos:
- script.js
- index.html
